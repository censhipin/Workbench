# 第四章：模块职责 (Module Responsibility)

## 4.1 模块总览

```
app/page.tsx          → UI 状态管理 + 事件调度（不包含业务逻辑）
lib/nlu/              → 自然语言理解（AI + 规则降级）
lib/v2/               → V2 执行引擎（编译 → 校验 → 执行 → 验证）
lib/execution-engine.ts → 顶层执行入口（调度 V2 + 构建 UI 展示数据）
lib/data-engine.ts    → 核心数据处理函数（无状态纯函数）
lib/audit-engine.ts   → 数据审计检测（无状态纯函数）
lib/file-engine.ts    → 文件解析和导出
lib/db.ts             → IndexedDB 持久化
lib/types.ts          → 全局类型定义
lib/pipeline-trace.ts → 调试追踪（全局单例）
components/           → UI 展示组件（纯渲染，不含业务逻辑）
```

## 4.2 各模块详细职责

### 4.2.1 app/page.tsx — UI 状态管理

**负责：**
- 管理所有 UI 状态（files, versions, executionSteps, activeTab 等）
- 调用 `parseIntentWithAI()` → 意图解析
- 调用 `executeIntent()` → 执行 + 动画 + 结果存储
- 调用 `createVersion()` → 版本创建
- 调用 `handleExport()` → 导出
- 显示错误弹窗 (`errorDialog`)
- 处理歧义确认 (`handleConfirmAmbiguity`)

**不负责：**
- 任何业务逻辑（筛选、排序、计算等）
- 修改数据（所有数据操作委托给下层）
- 列名解析或概念映射

### 4.2.2 lib/nlu/ — 自然语言理解

**负责：**
- 将自然语言 prompt 解析为结构化 TaskIntent
- AI 路径：调用 DeepSeek API → 解析 JSON → 编译 ExecutionPlan
- 规则降级路径：RuleBasedSemanticParser 操作识别 → 条件提取 → 列名解析 → 编译
- 列名概念映射（SchemaResolver）
- 采样数据提供给 AI（getSampleRows）

**不负责：**
- 任何数据执行
- 任何 UI 渲染
- 存储执行结果

### 4.2.3 lib/v2/ — V2 执行引擎

**子模块划分：**

| 子模块 | 职责 | 文件 |
|--------|------|------|
| task-compiler | TaskPlan → ExecutionPlan（columnHint → columnKey, operator 枚举转换） | task-compiler.ts |
| execution-engine | ExecutionPlan 调度执行（validate → exec → output → verify） | execution-engine.ts |
| executors/ | 各操作的实际执行逻辑（每个操作为一个类） | executors/*.ts |
| output-processor/ | 统一输出约束处理（includeColumns/limit 等） | output-processor/ |
| verifier/ | 执行结果验证（行数、完整性等） | verifier/*.ts |
| plan-validator | ExecutionPlan 结构与值类型校验 | plan-validator.ts |
| field-resolver | TaskPlan columnHint → 列名的概念匹配 | field-resolver.ts |
| execution-snapshot | 输入数据快照与输出数据克隆 | execution-snapshot.ts |
| predicate | 统一条件求值引擎（Operator 枚举 + cell value） | predicate.ts |
| types | Operator 枚举、ConditionExpr 接口 | types.ts |

**负责：**
- 编译：中文列名 → 机器 key，操作符字符串 → Operator 枚举
- 执行：通过 ExecutorRegistry 分发到具体 Executor
- 输出约束：统一的列筛选/重命名/排序/限制处理
- 结果验证：数据完整性校验
- 数据隔离：输入快照 + 输出克隆

**不负责：**
- 自然语言理解
- UI 渲染
- 持久化

### 4.2.4 lib/execution-engine.ts — 顶层执行入口

**负责：**
- 接收已完成解析的 TaskIntent（要求 v2plan 已存在）
- 前置校验（intent/operation/v2plan/sheet 存在性）
- 调用 runExecutionPlan() 执行 V2 计划
- 构建 5 步 PlanStep[] 用于 UI 展示
- 构建错误消息（buildErrorMessage）

**不负责：**
- 意图解析（那是 nlu/ 的事）
- 直接调用具体 Executor（那是 v2/execution-engine 的事）

### 4.2.5 lib/data-engine.ts — 核心数据处理

**所有函数都是无状态纯函数：**
- aggregateRows() — 分组聚合
- sortRowsMulti() — 多列排序
- filterRowsMulti() — 多条件筛选（支持 AND/OR）
- dedupRows() — 行级去重
- matchMultiTables() — 多表左连接匹配
- mergeTables() — 纵向合并
- cleanData() — 数据清洗

**负责：**
- 实际的数据行操作
- 始终返回新对象，不修改输入

**不负责：**
- 验证输入合法性（验证由 Executor 调用方负责）
- 操作符映射（Operator 枚举由 predicate.ts 消费）

### 4.2.6 lib/v2/predicate.ts — 条件求值

**负责：**
- `evaluateCondition(cellValue, operator, compareValue) → boolean`
- 字符串/数值/空值/集合/范围 五种条件类型的统一求值

**不负责：**
- 列名解析
- 数据类型推断
- 异常值 fallback

### 4.2.7 lib/audit-engine.ts — 数据审计

**负责：**
- 统计信息（行数、空单元格、空行）
- 重复行检测
- 空值检测（全列扫描）
- 异常检测（手机号/身份证/邮箱/日期/金额格式）
- 质量评分
- 修复建议生成
- 自动修复

**不负责：**
- 修改原始数据（autoFixRows 返回新数组，不修改输入）

### 4.2.8 lib/file-engine.ts — 文件处理

**负责：**
- parseFile(): Excel 文件 → WorkbenchFile（xlsx 库解析）
- exportToExcel(): 数据 → 浏览器下载

### 4.2.9 lib/db.ts — 持久化

**负责：**
- IndexedDB 读写
- 存储: 文件数据、任务文件 ID 列表、历史记录

**不负责：**
- 执行状态、版本数据、审计报告的持久化

## 4.4 职责边界（哪些不该做）

| 模块 | 不该做什么 |
|------|-----------|
| page.tsx | 不应包含业务逻辑（只做状态管理 + 事件转发） |
| semantic-parser.ts | 不应解析列名（那是 SchemaResolver 的事）；不应改数据 |
| task-compiler.ts | 不应做数据操作；不应推断列名语义（那是 field-resolver 的事） |
| execution-engine.ts | 不应处理自然语言；不应做 UI 相关操作 |
| executors/*.ts | 不应做数据验证（那是 verifier 的事）；不应做列筛选（那是 output-processor 的事） |
| data-engine.ts | 不应做操作符映射（那是 predicate.ts 的事） |
| components/*.tsx | 不应直接调用执行引擎或修改业务数据 |

## 4.5 当前职责断层

| 问题 | 表现 | 应该谁管 |
|------|------|---------|
| `task-compiler.ts` 的 `resolveColumn()` 做了两件事：精确匹配 + 模糊匹配 | 模糊匹配的阈值不明确 | 模糊匹配应归 `field-resolver.ts` |
| `execution-engine.ts` 的 `PlanStepBuilder` 在 execution-engine.ts 中 | 这是 UI 展示逻辑，不应该在执行层 | 应移到 components/ 或 page.tsx |
| `ruleIntentToTaskPlan` 和 `taskPlanToIntent` 是双向转换但有信息丢失（如 `logic: 'OR'`） | 过去曾丢失 OR 条件 | 转换器应该维护完整的双向映射 |
| `AuditEngine` 的空值检测和 `CleanExecutor` 的空行删除没有共享逻辑 | 两者各自实现自己的空值判定 | 应共享同一个 `isNull()` / `isEmptyRow()` 函数 |
| `FormulaExecutor` 的 `computeCell()` 是一个 130 行的 switch，包含文本/日期/数值/IF 多种逻辑 | 模块内聚性差 | 每种计算类型应有独立函数 |
