这些问题是 **本次改动未触及的**。当用户没有 DeepSeek API Key 时走规则降级路径，大量测试用例失败。

| 问题 | 受影响用例 | 根因 | 文件 |

| **"高于"不被识别为"大于"** | 基础筛选 | `semantic-parser.ts` 的同义词库缺"高于"、"超过"等 | `semantic-parser.ts` |

| **规则 pipeline 分割后 sort 缺 columnHint** | T13 及其他 pipeline | `parsePipelineSteps` 用正则切割后 sort 段没提取到列名 | `semantic-parser.ts` line 716 |

## 四、关键技术决策

### 架构约束（不要改）
- `lib/v2/` 下的 V2 框架核心（ExecutorRegistry, ExecutionEngine, OutputProcessor）不改
- 新增操作类型只需注册 Executor，不动现有执行引擎
- 所有数据在客户端处理，不上传

### 可选的改进方向
1. **弱化规则解析器**：如果大多数用户都会配 API Key，可以适当降低规则解析的覆盖要求，只保证最基础的筛选/排序即可
2. **本地模型替代**：用 Ollama 等本地模型替代规则降级方案，NLU 质量会大幅提升
